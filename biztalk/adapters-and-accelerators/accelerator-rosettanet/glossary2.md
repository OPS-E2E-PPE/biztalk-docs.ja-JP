## <a name="d"></a>D  
 **Data Universal Numbering System (D-U-N-S) の数**  
 ビジネスの所在地を識別し、グローバルな範囲で一意にシーケンシャルに生成される 9 桁の数。  
  
 **配信ヘッダー**  
 RosettaNet メッセージの一部です。 Delivery header は、メッセージの送信者、受信者、およびメッセージ インスタンスの情報を識別する XML ドキュメントです。  
  
 **送信先組織**  
 ドキュメントの送信先としてでは、メッセージング ポートが指定されている組織。  
  
 **ダイジェスト アルゴリズム**  
 入力としてメッセージを受け取り、ハッシュや、きわめて複雑な方法で、メッセージの内容に依存するビットの固定長の一連のダイジェストを生成するアルゴリズム。 設計条件には、非常に困難になるため、ダイジェストの偽造や、ダイジェストを変更することがなく、メッセージを変更するすべてのユーザーが含まれます。 通常、ダイジェスト アルゴリズムを使用するアプリケーションはメッセージの認証とデジタル署名のスキームです。 広く使用されているアルゴリズムには、MD5 と SHA1 が含まれます。 BTARN では、送信メッセージの両方の MD5 と SHA1 を受信メッセージと SHA1 のみをサポートします。  
  
 **ドキュメント定義**  
 特定のドキュメントを表すプロパティのセット。 ドキュメント定義のプロパティは、ドキュメント仕様へのポインターを含めるし、グローバル追跡フィールドと選択条件を含めることができます。  
  
 **ドキュメント インスタンス**  
 送信される実際のデータの表現を[!INCLUDE[btsBizTalkServerNoVersion](../../includes/btsbiztalkservernoversion-md.md)]します。 ドキュメント インスタンスは、ドキュメント インスタンス構造体に含まれている特定のデータの表現であるときに、仕様は、データの構造を定義で、ドキュメント仕様とは異なります。  
  
 **ドキュメント型定義 (DTD)**  
 その他の要素と属性に存在する要素と属性を指定する標準の定義と、順序、頻度、およびコンテンツに対する制約を指定します。  
  
 **ダブル アクション トランザクション**   
 イニシエーターが、要求アクションを送信プロセスでは、応答アクションの後に、応答側からシグナルを受信します。 イニシエーターは、応答アクションにシグナルを送信することによって、プロセスを終了します。  
  
## <a name="e"></a>E  
 **拡張マークアップ言語 (XML)**  
 標準の HTML の機能のカスタマイズされたタグを作成することによって、World Wide Web Consortium (W3C) によって開発された仕様。 HTML では、ページ内の要素を記述する定義済みタグのみを使用して、XML は、ページの開発者によって定義されるタグを使用できます。 製品や支払金額など、事実上すべてのデータ項目のタグは、特定のアプリケーションで使用できます。 これにより、データベースのレコードとして機能する Web ページです。  
  
 **Extensible Stylesheet Language (XSL)**  
 XML ドキュメントのスタイル シートの書式。 XSL を使用して、HTML の表示を定義するカスケード スタイル シート (CSS) が使用されるのと同じように、XML の表示を定義します。 [!INCLUDE[btsBizTalkServerNoVersion](../../includes/btsbiztalkservernoversion-md.md)] 2 つの仕様間の変換言語として XSL を使用します。

## <a name="p"></a>P  
 **パッケージ化**  
 XML 表現、またはその逆に、RosettaNet メッセージを変換するプロセス。  
  
 **Partner Interface Process (PIP)**  
 PIP では、一連のビジネス ドキュメントおよびドキュメント コンテンツの詳細を含む、契約の詳細について説明します。  
  
 **PIP 仕様ドキュメント**  
 BTARN 管理コンソールでプロセス構成を作成するときに使用する設定についてのガイダンスを含むドキュメント。 RosettaNet.org からダウンロードした、RosettaNet 組織から PIP 仕様ドキュメントと PIP をダウンロードします。BTARN 管理コンソールでプロセス構成を作成するときに使用する設定についてのガイダンスを含むドキュメント。 RosettaNet.org からダウンロードした、RosettaNet 組織から PIP 仕様ドキュメントと PIP をダウンロードします。  
  
 **preamble のヘッダー**  
 名前とビジネス メッセージが準拠する標準のバージョンを識別する XML ノード。 完全な RosettaNet メッセージを形成する他のヘッダーと共にパッケージ化されます。 Preamble とも呼ばれます。  
  
 **プライベート プロセス**   
 組織の内部ビジネス プロセス。 BTARN は、長時間実行される BizTalk オーケストレーションとして、プライベート プロセスを実装します。  
  
 **プロセス構成設定 (PCS) プロファイル**  
 パートナー アグリーメントを実行する方法を決定します。 PCS プロファイルを使用すると、構成の詳細の RosettaNet PIP Partner Interface Process () を入力します。 PCS プロファイル内の 1 つの要素に RosettaNet PIP 仕様のマップで指定されたすべての構成値。 複数のパートナー アグリーメントの 1 つの PCS プロファイルを使用できます。  
  
 **port**  
 特定の実装を使用する名前付きの場所。 BizTalk オーケストレーション デザイナーでは、ポートがするメッセージの送信先またはメッセージの受信元となる場所と通信アクションを実装するために使用されるテクノロジによって定義されます。 場所は、ポートの名前によって一意に識別します。  
  
 **パブリック プロセス**   
 パブリック プロセスとして取引先との統合に関係するビジネス プロセス。 BTARN では、長時間実行される BizTalk オーケストレーションとしてパブリック プロセスを実装します。 1 つのパブリック プロセス オーケストレーションは、発信側と応答側のいずれかで実行されます。 BTARN セットアップ プログラムでは、発信側と RNIF 1.1 および RNIF 2.01 の応答側パブリック プロセス オーケストレーションのバージョンを提供します。 これらのパブリック プロセス オーケストレーションは、すべての RNIF プロセスを実装します。 パブリック プロセスには、残りのコンポーネントから RNIF の複雑さが非表示にします。 RNIF 準拠のメッセージ フローをするだけでなく、パブリック プロセスも既定の追跡設定を決定し、実行時にプロセス状態に関する情報を提供します。

 **サービスのヘッダー**  
 PIP、ビジネス アクティビティ サービスでは、取引先のパートナー、およびロールの送受信の動作など、ビジネス メッセージに関連付けられている部分を識別する XML ドキュメント。  
  
 **シグナル URL**  
 ホーム組織がシグナル メッセージを送信する URL。 たとえば、 http://FabrikamServer/BTARNApp/RNIFReceive.aspx のようにします。  
  
 **シングル アクション通知**  
 イニシエーターが 1 つのアクション メッセージと応答側が送信プロセスは、メッセージで応答します。  
  
 **仕様**  
 A [!INCLUDE[btsBizTalkServerNoVersion](../../includes/btsbiztalkservernoversion-md.md)]-特定の XML スキーマ。 BizTalk エディターでの仕様を作成し、区切り形式、位置指定、または区切り形式、および位置指定などに、EDIFACT、x12、および XML、または、フラット ファイルなどの業界標準に基づくことができます。 BizTalk マッパーでは、仕様、送信元仕様と送信先仕様として開かれてを使用して、マップを作成します。  
  
 **同期 URL**  
 ホーム組織が、パートナーとの同期トランザクションを確立するために使用する URL http://FabikamServer/BTARNApp/RNIFReceive.aspx です。  
  
 **同期トランザクション**  
 イニシエーターを返す応答 (ダブル アクション) またはシグナル (シングル アクション) 同じ HTTP 状態の接続を閉じることがなく処理します。

 **トランザクション**

## <a name="v"></a>V  
 **検証アダプター**  
 検証アダプター インターフェイスを実装するアプリケーション。 パブリック応答側は、アクション メッセージ (要求または応答) を受信すると、検証アダプターを呼び出します。 任意の受信メッセージを受け入れる前に、ビジネスを必要とする検証規則のセットに含めることができます。 BTARN は、受信パイプラインとパブリック オーケストレーションでネイティブに検証を実行します。
