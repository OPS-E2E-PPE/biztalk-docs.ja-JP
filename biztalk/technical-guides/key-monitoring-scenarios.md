# <a name="key-monitoring-scenarios"></a>主要な監視シナリオ
[!INCLUDE[btsBizTalkServerNoVersion](../includes/btsbiztalkservernoversion-md.md)] Operations Manager 2007 R2 または 2012 用管理パックには、次の表で説明されている主要な監視シナリオの数値が含まれています。  
  
### <a name="biztalk-server-monitoring-scenarios"></a>BizTalk Server の監視シナリオ  
  
|シナリオ|説明|  
|--------------|-----------------|  
|保留メッセージ警告|監視および中断されたメッセージに関連する問題の解決は、BizTalk Server 環境での一般的な運用タスクには。 BizTalk Server 管理パックを使用すると、監視し、合理化された方法で保留メッセージ イベントをトラブルシューティングできます。 メッセージの保留の 2 つの種類は、メッセージを処理するときに BizTalk Server で発生します。受信メッセージの中断と送信メッセージの保留します。<br /><br /> 受信メッセージの処理エラーやサブスクリプションの欠落などのさまざまな種類にメッセージを中断することがあります。 メッセージを中断または、メッセージを保留する一部のトランスポートを構成できます。 送信メッセージの処理エラーをすべては、メッセージが保留に常につながります。<br /><br /> 受信メッセージは、中断のアラート ルールがアダプターごとに作成されます。 このアプローチより柔軟に、受信側でメッセージ保留の処理を提供します。 通知など、アダプターに応じて異なるレスポンス アクションを追加することができます。 さらに、カスタム ルールを作成して、BizTalk Server 管理パックをカスタマイズできます。 送信メッセージは、すべてのメッセージが常に任意の種類の送信時のエラーで中断されます。|  
|アラートの抑制ポリシー|中断されたメッセージのルールは、他の規則が適用された通常よりも、さまざまな抑制ポリシーを使用します。 警告を抑制警告名、アラートのソース、およびコンピューターとドメインのベースします。<br /><br /> 抑制は警告ソースに基づいて、各保留メッセージ ルールのアラートのソースに保留メッセージ イベントから抽出されたパラメーターも含まれます。 このパラメーターは、メッセージが受信または送信される URI です。<br /><br /> 同じ URI で発生したエラーは、通常、同じ理由。 そのため、複数の関連する障害イベントが存在する場合でも、同じの根本原因の 1 つのアラートを作成することをお勧めします。 これにより、1 つのアラートと 1 つの根本原因を用意することが発生したときの問題を修正が簡単です。 、Uri に基づいて警告を整理は、不要なアラートを減らすし、未解決の問題を効率的に管理を容易にします。 メッセージの保留の uri でない過去に保留警告の新しいアラートが作成されます。|  
|BAM の技術的なサポートのアラート|通常、ビジネス アクティビティ監視 (BAM) ポータルのユーザーは、ビジネス ユーザーと呼ばれるには、ビジネス レベルのインジケーターとビジネス アクティビティが理解します。 さらに詳しく調査するモニター集計インジケーター、包括的なプロセスでビジネス トランザクション、または特定のプロパティを持つトランザクションを検索する BAM ポータルを使用します。 ただし、ビジネス ユーザーは多くの場合、IT インフラストラクチャ レベルの問題の側面を理解できません。 特定のトランザクションの調査、中に BAM ポータルではビジネス ユーザーが IT インフラストラクチャ レベルでトランザクションを調査する IT 運用スタッフから技術的なサポートを要求します。<br /><br /> BAM ポータルは、このような技術的なサポート要求を通じて、ビジネス ユーザーから使用できます。 技術的なサポートを要求して、BAM ポータルがホストされているコンピューターのイベント ビューアーのアプリケーション ログにエントリを作成します。 このイベントには、技術的なサポートがユーザーとメッセージ インスタンス識別子、サービス インスタンス識別子、および BizTalk Server グループの情報などの他の重要な情報を要求したアクティビティの詳細が含まれています。<br /><br /> BizTalk Server 管理パックには、BAM の技術的なサポート イベントの検出時にアラートをトリガーするルールが含まれています。 このアラートを作成するこのルールは、"エラー。BAM Technical assistance Required.。" BAM ポータルから送信された各要求では、新しいアラートが作成されます。|  
|BizTalk メッセージ ボックスおよびホスト|BizTalk Server 管理パックには、データベースやキューは、BizTalk メッセージ ボックスの状態を包括的なビューを提供するパフォーマンスしきい値ルールが組み込まれています。 しきい値規則の 2 種類が用意されています。<br /><br /> -に適用される規則一般的に、つまり、すべての BizTalk ホストでは、すべての BizTalk メッセージ ボックス データベース。<br />-特定の BizTalk ホストまたはメッセージ ボックスに固有の規則。<br /><br /> 汎用的なルールは、すべての BizTalk Server ホストまたは共通のしきい値に基づくメッセージ ボックス データベースを監視します。 たとえば、"Monitor HostQ Size"ルールをすべての BizTalk Server ホストの作業キューに基づいて、共通のしきい値。 3 つのホストが存在しない場合は、同じルールによって、すべての作業キューが監視され、ホストの作業キューのいずれかの共通のしきい値を越える場合に警告が発生しました。<br /><br /> BizTalk Server ホストに固有のルールでは、さまざまなホストとメッセージ ボックス データベースを別のしきい値を構成できます。 たとえば、"Monitor HostQ Size – BizTalkServerApplication"ルールは、BizTalkServerApplication ホストの作業キューを監視するホスト固有の規則です。 これには、特定のパフォーマンス カウンター インスタンスの特定の Operations Manager プロバイダーを定義してしきい値規則でそのプロバイダーを使用しています。<br /><br /> ホストに固有のルールは、環境内で適用可能なルールを作成するためのガイドとして使用するテンプレート規則に提供されます。 汎用的なルールは、環境に固有のしきい値の値で構成される必要があります。 BizTalk ホスト/メッセージ ボックス固有のルールは、テンプレート ルールと適切なしきい値に基づいて作成する必要があります。<br /><br /> BizTalk Server が組み込まれています、自己制限各種パラメーターに基づいてオーバー ロードを防ぐことができます。 スロットルが発生する原因となる一時的なオーバー ロードは、運用上あまり大きなイベントではありません。 永続的な調整、ただし、安定した環境がないと、インフラストラクチャ レベルの基になる問題がある可能性があります。 BizTalk Server 管理パックは、次の条件に基づくパフォーマンスしきい値モニタで、このような永続的な制限の条件のプロアクティブな監視を提供します。<br /><br /> BizTalk Server サービス プロセスのメモリ<br />-処理されるメッセージの数<br />-BizTalk Server プロセス内のスレッドの数<br />BizTalk データベース キューのサイズ|  
|オーケストレーション エラー|中断されたオーケストレーションの監視は、もう 1 つの一般的なシナリオです。 オーケストレーションは、オーケストレーション エンジンの問題、.NET コンポーネントの障害、およびプログラム ロジックを含む多くの理由により失敗します。 問題の原因を迅速に判断し、解決されると、中断されたオーケストレーションを再開するオペレーターを有効にするのには重要です。|  
|サービスの監視|BizTalk Server では、正常に動作する多数のサービスが実行されている必要があります。 これらのサービスの状態は、別の BizTalk サービスを表すクラスのモニターにマップされます。|  
|アプリケーションの監視|BizTalk Server には、ソリューションの成果物をグループ化するため、アプリケーションの概念が導入されました。 アプリケーションの監視は、アプリケーション レベルのエラーの影響を検出します。|
