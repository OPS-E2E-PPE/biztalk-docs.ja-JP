# <a name="verification-phase-recommendations"></a>検証フェーズの推奨事項
システムが完全なコードの後は、完全に安定する準備ができてとリリース条件を確認することができます。 このフェーズは、安定化フェーズとも呼ばれます。 このフェーズの最終目標は、特定してバグを修正し、システムが実稼働の準備ができていることを証明します。 このフェーズでは、システムのリリース候補のテストの最終ラウンドそのため、必要があります。  
  
 リリース候補は完了と見なされるシステム (通常は、最新) のバージョンであり、リリース バージョンに十分な安定性は、検証テストに合格します。 これが実証されている方法、銀行の機能が正常に完了、パフォーマンス、およびストレス テスト実際の準備ができたことを確認することです。  
  
## <a name="test-to-verify-sustainable-throughput-and-latency"></a>維持可能なスループットと待機時間を確認するテスト  
 パフォーマンスのテストの検証は、実装フェーズと並列で開始されましたが、リリース条件テストの完全なセットに正常に耐え得るも示されているリリース候補を確定する必要があります。 最適な状態、リリース候補の変更は行われません、最終テスト段階中に高の回帰が生じていないことです。 実際には、これはとても困難ですし、回帰のリスクに関する評価を行う必要があるように変更は、ビルドにチェックインされる、します。  
  
 たとえば、パイプラインやオーケストレーションなどの成果物をシステムに根本的な変更を導入すると、パフォーマンス テストを順序で再実行する必要がありますはこの新しいリリース候補を検証します。  
  
 システムが実稼働の準備ができていることを確認するには、持続可能な方法のエンド ツー エンドでテストされていることを確認する必要があります。 つまり、すべての操作アクティビティなどのデータベース メンテナンス、クエリを実行して、計画的な操作と計画外の停止をテストする必要があります、トピックで定義されている[維持可能なパフォーマンスとは何ですか?](../core/what-is-sustainable-performance.md) これは、最終テスト段階で維持可能なパフォーマンス テストのスイート全体を結合することが重要であるため、システムの準備ができてを証明する最後のチャンスです。  
  
## <a name="identify-bottlenecks-and-adjust-hardware-or-solution-to-remove-goal-blockers"></a>ボトルネックを特定し、目標ブロッカーを削除するには、ハードウェアまたはソリューションを調整します。  
 実際が、最終テスト段階のテスト ベッドはテスト ベッドの開発よりもハードウェアに関して、運用環境に近いことが一般的です。  そのため、最終的なテスト パス中に営業案件を使用して、システムで、新規または既存のボトルネックを特定し、ハードウェアの調整が必要に十分な大きさのかどうかは、決定することが重要です。 貴重な計画および運用情報場合でも、ハードウェアは、すぐに調整する必要がある、最も一般的なシステムのボトルネックを識別する提供されます。  
  
 たとえば、システム sustains 実稼動ロード プロファイルで、メッセージ ボックス サーバー上の物理ディスクのアイドル時間が (たとえば、20%) の下に不足しているが観察された場合は、実稼動中にこのディスクを監視として識別できます、キーの正常性インジケーター。 さらに、システムの読み込み機能のためのプランでは、ディスク サブシステムを向上させる必要があるナレッジできるようになりました。
